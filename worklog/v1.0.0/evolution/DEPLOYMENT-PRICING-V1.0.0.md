# DTS 部署模式与硬件定价方案 V1.0.0

> Decision Twins System — Deployment & Hardware Pricing Strategy
> 软硬一体交付，覆盖中小企业到政府信创全场景

---

## 1. 战略背景

### 1.1 市场洞察

在中国及部分亚太市场，企业软件的销售存在一个根深蒂固的认知：

- **软件无形，硬件有形** — 客户更愿意为"看得见摸得着"的硬件付费
- **预算科目限制** — 很多企事业单位有"设备采购"预算但没有"软件采购"预算
- **国产化/信创要求** — 政府、央企、军工必须采购国产软硬件

因此，DTS 提供两种交付模式：

| 模式 | 说明 |
|------|------|
| **纯软件** | 客户自备硬件/云资源，DTS 提供软件授权 + 部署服务 |
| **软硬一体** | DTS 预装在认证硬件上，整机交付，开箱即用 |

本文档聚焦**软硬一体**模式的方案设计。

### 1.2 设计原则

1. **开箱即用** — 通电联网即可使用，Day 1 部署时间 < 1 小时
2. **平滑升级** — 一体机 → 集群的迁移路径清晰，数据零丢失
3. **灵活定价** — 支持买断/订阅/整机打包三种报价方式，适配不同投标场景
4. **信创兼容** — 国产 CPU(鲲鹏/飞腾) + 国产 GPU(昇腾) + 国产 OS(麒麟/统信)
5. **硬件合作开放** — 不绑定特定厂商，BOM 清单认证制

## 2. 一体机模式 (DTS-Box)

### 2.1 产品定位

单台服务器承载 DTS 全栈，适用于：
- 5-50 人团队
- 无专职 IT 运维人员
- 物理隔离/涉密网络
- 预算 ≤ 50 万

### 2.2 硬件形态

| 形态 | 适用场景 | 说明 |
|------|---------|------|
| **塔式工作站** | 中小企业，放办公室 | 安静、紧凑，无需机房 |
| **1U/2U 机架式** | 政府/涉密，放机柜 | 信创认证常见形态 |

两种形态软件栈完全一致，切换成本为零（仅硬件外壳差异）。

### 2.3 三档 SKU

#### DTS-Box Lite — 入门款（¥10万）

| 项目 | 配置 |
|------|------|
| **CPU** | 8C16T+ (x86/ARM) |
| **内存** | 64 GB |
| **存储** | 1TB NVMe SSD |
| **GPU** | 无（纯 CPU 推理或外接 LLM API） |
| **模型能力** | 7B 量化模型 / 云端 LLM 代理 |
| **并发用户** | 5-15 人 |
| **K8s** | K3s 单节点 |

**典型场景：** 预算有限的中小企业试点，或作为边缘节点接入中心集群。

| 项目 | 配置 |

| **内存** | 128 GB |

| **并发用户** | 15-50 人 |

**典型场景：** 中小企业主力款，离线环境完整运行 DTS 全栈 + 本地 LLM。

#### DTS-Box Gov — 信创款（¥45万）

| 项目 | 配置 |
|------|------|

| **GPU** | 昇腾 310B / 910B |
| **OS** | 麒麟 V10 / 统信 UOS |
| **模型能力** | 14B-70B（视昇腾型号） |
| **并发用户** | 15-50 人 |
| **K8s** | K3s (ARM64) 单节点 |

**典型场景：** 政府、央企、军工涉密场景，满足国产化替代要求。

## 3. 三节点集群模式 (DTS-Cluster)

### 3.1 产品定位

三台服务器组成高可用集群，适用于：
- 100+ 并发用户
- 生产环境不允许停机
- 大数据量（TB 级）
- 需要独立 GPU 算力

### 3.2 集群架构

```
┌─────────────────────────────────────────────────┐
│                  K8s HA Cluster                  │
│              (etcd 3 副本，任一节点可选主)         │
│                                                  │
│  ┌──────────────┐  ┌──────────────┐             │
│  │  Node 1      │  │  Node 2      │  ← 业务节点 │
│  │  (业务+管理)  │  │  (业务+管理)  │    互为备份  │
│  │              │  │              │             │
│  │ Java 10 svc  │  │ Java 10 svc  │             │
│  │ Go 2 svc     │  │ Go 2 svc     │             │
│  │ Frontend 3   │  │ Frontend 3   │             │
│  │ 中间件(HA)   │  │ 中间件(HA)   │             │
│  └──────────────┘  └──────────────┘             │
│                                                  │
│  ┌──────────────────────────────┐               │
│  │  Node 3 (GPU 节点)           │  ← 独立算力   │
│  │                              │    可按需升级  │
│  │  Python AI 10 svc            │               │
│  │  LLM 推理 (vLLM/Ollama)     │               │
│  │  GPU: L40S/A800/昇腾910B    │               │
│  └──────────────────────────────┘               │
└─────────────────────────────────────────────────┘
```

**架构特点：**
- **Node 1 + Node 2：** 对称互备，运行 Java/Go/Frontend 服务 + HA 中间件
- **Node 3：** 独立 GPU 节点，运行 Python AI 服务 + LLM 推理
- 任一业务节点故障，另一台自动接管（K8s 调度）
- GPU 节点独立升级不影响业务连续性
- 保留向对称三节点/5-10 节点扩展的迁移路径

### 3.3 两档 SKU

#### DTS-Cluster Standard — 标准集群（¥100-200万）

| 节点 | 配置 | 数量 |
|------|------|------|
| **业务节点** | 32C64T, 256GB RAM, 4TB NVMe | ×2 |
| **GPU 节点** | 32C64T, 256GB RAM, 2TB NVMe, NVIDIA L40S 48GB 或 A800 80GB | ×1 |

| 项目 | 规格 |
|------|------|
| **并发用户** | 100-500 |
| **数据规模** | TB 级 |
| **模型能力** | 70B+ 模型，高速推理 |
| **HA** | 业务节点互备，中间件主备/多副本 |

#### DTS-Cluster Gov — 信创集群（¥200-400万）

| 节点 | 配置 | 数量 |
|------|------|------|
| **业务节点** | 鲲鹏920 32C, 256GB, 4TB 国产SSD | ×2 |
| **GPU 节点** | 鲲鹏920 + 昇腾910B, 256GB, 2TB | ×1 |
| **OS** | 麒麟 V10 / 统信 UOS |
| **认证** | 全栈信创目录认证 |

## 4. 定价策略

### 4.1 价格总览

| SKU | 报价 | 含首年服务 |
|-----|------|-----------|
| DTS-Box Lite | **¥10万** | ✓ |
| DTS-Box Pro | **¥30万** | ✓ |
| DTS-Box Gov | **¥45万** | ✓ |
| DTS-Cluster Standard | **¥100-200万** | ✓ |
| DTS-Cluster Gov | **¥200-400万** | ✓ |

### 4.2 报价组合方式

根据投标场景灵活选择：

```
方式 A — 一次性买断
├── 硬件采购费
├── 软件永久授权
├── 首年实施 + 维保
└── 次年起维保费 (报价 15-20%)

方式 B — 硬件 + 订阅
├── 硬件按成本价（甚至补贴）
├── 软件年费订阅 (3-5 年锁定)
├── 首年实施费
└── 年费含维保 + 升级

方式 C — 整机方案价
├── 不拆分硬软件
├── 按"AI 决策解决方案"整体报价
├── 利润内含，简化采购流程
└── 适合预算科目为"设备采购"的客户
```

### 4.3 价格构成拆解（内部核算）

## 5. 架构迁移路径

### 5.1 升级路径图

```
DTS-Box Lite ───── 加装 GPU ────→ DTS-Box Pro
    │                                  │
    │    ┌─── 信创替换 ───→ DTS-Box Gov │
    │    │                              │
    └────┴──── 数据迁移 ────→ DTS-Cluster Standard
                                       │
                              信创替换 ──→ DTS-Cluster Gov
                                       │
                              加节点 ───→ 5-10 节点大集群
```

### 5.2 迁移保障

| 能力 | 实现方式 |
|------|---------|
| **数据迁移** | `dts-cli migrate` — 全量备份 + 增量同步 |
| **配置迁移** | dts-operator CRD 导出/导入 |
| **AppPack 迁移** | Pack 跨部署模式通用，无需重新开发 |
| **零停机迁移** | 集群模式下支持滚动迁移 |
| **回滚** | 迁移前自动快照，失败可一键回滚 |

## 6. 交付流程

### 6.1 一体机交付

```
Day 0: 硬件出厂预装 DTS + 质检
        ↓
Day 1: 到场通电 → 系统初始化向导(30min)
       → 接入客户 1-2 个核心数据源
        ↓
Day 2-3: 配置行业 AppPack → 导入业务知识
        ↓
Day 4-5: 用户培训 + 验收
```

### 6.2 集群交付

```
Day 0:    硬件到场上架
Day 1:    K8s 集群初始化 (dts-cli install)
Day 2:    DTS 全栈部署 + HA 验证
Day 3-5:  数据源接入 + AppPack 配置
Day 6-8:  压力测试 + HA 故障演练
Day 9-10: 用户培训 + 验收
```

## 7. 硬件合作模式（待定）

硬件合作模式暂不锁定，后续根据市场节奏选择：

| 模式 | 优势 | 劣势 | 适用阶段 |
|------|------|------|---------|
| **OEM 贴牌** | 品牌统一，利润高 | 库存风险，资金压力大 | 规模化阶段 |
| **渠道合作** | 轻资产，借力厂商渠道 | 利润分薄，品控弱 | 快速扩张阶段 |
| **BOM 认证** | 最灵活，零库存 | 交付一致性难控 | 早期验证阶段 |
| **混合模式** | 按客户需求灵活组合 | 管理复杂 | 成熟阶段 |

**当前策略：** 先以 BOM 认证清单模式启动，积累交付案例后再考虑 OEM/渠道合作。

## 8. 与现有定价体系的关系

本方案是 BP 中三层定价模型的**硬件延伸**：

```
现有三层定价 (纯软件)          硬件捆绑定价 (本方案)
┌─────────────────┐          ┌─────────────────────┐
│ L1: 平台基础费   │    →     │ 内含在整机报价中      │
│ L2: AppPack 费  │    →     │ 可选增值，独立报价     │
│ L3: 知识分润    │    →     │ 不变，客户知识资产分润  │
└─────────────────┘          └─────────────────────┘

纯软件客户: L1 + L2 + L3
硬件客户:   整机价(含L1) + L2(可选) + L3
```
