## 1. 执行摘要

**Decision Twins System (DTS)** 是面向决策密集型行业的 **AI 决策操作系统**。

**核心公式**：**Ontology（认知基座） + Intent（意图理解） + Agent（自主执行）**

企业不缺数据，缺的是基于数据的决策能力。DTS 将专家多年积累的隐性经验，通过 AI 对话提取、结构化编码、系统化运行，让一位专家的 20 年经验能被 500 人使用。用户不再需要写 SQL、做报表、等排期，而是用一句话完成从"理解需求 → 查数据 → 分析 → 图表 → 报告 → 行动建议"的端到端决策闭环。

**关键差异化**：

| 维度 | 传统方案 | DTS |
|------|---------|-----|
| 与数据中台的区别 | 解决"有没有数据" | 解决"数据能不能做决策" |
| 与 BI 工具的区别 | 展示数据让人看 | 理解意图替人想、帮人做 |
| 与 AI Agent 平台的区别 | 通用 Agent，无行业深度 | Ontology + KnowHow，行业认知是一等公民 |
| 与 Palantir 的区别 | 重、贵、封闭，不服务中国市场 | 轻量私有化、开放生态、KnowHow 可交易 |

**目标市场**：能源（电网/石油/新能源）、科研院所、先进制造业等决策密集型行业。

**核心指标**：

- Y1 收入 800 万，Y3 收入 1 亿
- 57 个内置 AI 技能，25 个微服务，三语言协作
- DAP 五层协议构建生态护城河
- 五条铁律保障企业信任

**融资需求**：种子轮 2,000 万人民币，用于核心产品研发（60%）+ 标杆客户落地（25%）+ 运营储备（15%）。

## 2. 市场机会

### 2.1 企业决策的三大痛点

**痛点一：专家依赖症**

中国制造业、能源行业的核心决策高度依赖少数资深专家。一个经验丰富的电网调度员需要 8-10 年培养周期，一个高级质量工程师需要 5-8 年积累。这些专家的决策依据大部分存在于"脑子里"——波兰尼称之为"缄默知识"（Tacit Knowledge）：我们知道的比我们能说出来的多。

**痛点二：经验流失危机**

关键人员离职、退休导致企业核心竞争力随人走。据统计，中国制造业核心技术岗位年均流动率 15-20%，每流失一位资深工程师，企业需要 2-3 年重新培养替代者。知识归零、重复交学费，是企业长期面临的隐性损失。

**痛点三：决策质量不一致**

同一个公司，老手做的分析和新手做的分析，质量差距可达 10 倍。关键决策的质量取决于"今天谁值班"，而非组织能力。企业花数千万建的数据基础设施（数据湖、数仓、BI），最终产出仍取决于少数人的个人能力。

### 2.2 TAM / SAM / SOM 分析

**TAM（总体市场规模）**：中国企业 AI + 决策智能市场

- 企业 AI 应用市场：2026 年约 2,500 亿元，年增速 35%+（IDC 数据）
- 决策智能细分：约 800 亿元（含 BI 升级、AI Agent、行业智能决策）

**SAM（可服务市场）**：DTS 目标行业的决策平台需求

- 能源行业（电力/石油/新能源）：~120 亿元
- 科研院所（高校/科研机构/实验室管理）：~40 亿元
- 先进制造（装备/电子/汽车零部件）：~150 亿元
- 合计 SAM：~310 亿元

**SOM（可获取市场）**：DTS 三年内现实可获取

- Y1（种子期）：2-3 个标杆客户，800 万
- Y2（增长期）：8-12 个行业客户，3,500 万
- Y3（规模化）：20+ 客户 + AppPack 市场收入，1 亿
- 三年 SOM 渗透率：约 0.03%（极保守，验证商业模式阶段）

### 2.3 行业趋势：三个条件首次同时成熟

**AI Agent 能力就绪**：2024-2026 年大语言模型的推理、规划、工具调用能力突破临界点，Agent 从"玩具"变成"生产力工具"。Manus AI 8 个月达到 $100M ARR 并被 Meta $2B 收购，验证了 AI Agent 市场的真实需求和资本热度。

**企业数据就绪**：中国企业过去 5-10 年投入数百亿建设数据基础设施（数据湖、数仓、ETL、BI），但 ROI 普遍不达预期。这些投资构成了 AI 可用的数据基础，DTS 是"让已有投资增值"的最后一公里。

**客户意愿就绪**：企业决策者亲身体验了 ChatGPT/Manus 等 AI 产品的能力，首次主动寻求"AI + 业务决策"的落地方案，从被动接受 IT 提案转为主动推动。

## 3. 产品定位

### 3.1 Decision Twins — 决策过程的数字孪生

DTS 提出的"Decision Twin"概念，与工业界熟知的 Digital Twin（物理孪生）完全不同：

| 维度 | Digital Twin（物理孪生） | Decision Twin（决策孪生） |
|------|------------------------|------------------------|
| 孪生对象 | 物理设备/系统的结构和状态 | 专家的决策过程和判断逻辑 |
| 核心价值 | 模拟物理行为 | 复制决策能力 |
| 关键输入 | 传感器数据、CAD 模型 | 隐性知识、业务规则、行业经验 |
| 关键输出 | 设备状态预测 | 决策建议、行动方案、分析报告 |
| 技术基础 | IoT + 仿真引擎 | Ontology + AI Agent + KnowHow |

**一句话定义**：Decision Twin 是把企业的数据资产、业务规则、行业知识编织成一个能替决策者"想"的系统。不是替代人决策，而是让每个人都拥有专家级的决策能力。

### 3.2 与传统方案的本质区别

**vs 传统 BI**：BI 做的是"展示"——给你一张图，你自己想办法。DTS 做的是"执行"——你说一句话，系统端到端完成从查数到出报告到给建议的全流程。BI 的终点是"看图"，DTS 的终点是"行动"。

**vs 数据中台**：数据中台解决的是"有没有数据"的问题——数据搬运、清洗、存储。DTS 解决的是"数据能不能做决策"的问题。DTS 不搬数据，不替换数据中台，而是站在数据中台之上，补齐"最后一公里"。

**vs 通用 AI Agent**（Manus/Coze/Dify）：通用 Agent 是万金油，什么都能做但什么都不深。DTS 有 Ontology（行业认知图谱）和 KnowHow（行业隐性知识），是带着 20 年行业经验的专家 Agent，而非刚入职的通用助理。

### 3.3 三大产品理念

**理念一：人工随时可接管**

这是 DTS 五条铁律中的第一条。删掉所有 AI 组件，系统仍然 100% 可用。AI 是加速器，不是唯一通道。这不是技术妥协，而是深思熟虑的商业决策——企业决策者最怕的不是 AI 不够强，而是"万一 AI 出错怎么办"。DTS 的回答是：任何时候，任何步骤，人都可以一键接管。

**理念二：安全是基因，不是补丁**

数据安全不是功能模块，而是架构基因。所有数据出口必经安全检查点，AI 只看授权数据，查询前过滤而非查询后脱敏，全操作审计不可篡改。五条铁律在架构层面保障，不是靠运维纪律。

**理念三：能力先于界面**

API-first 原则。每个能力先有完整的 API + 测试 + 安全审计，再做 UI。不做"好看但跑不通"的 AI 界面。先闭环"理解 → 查数 → 分析 → 图表 → 报告 → Action"的能力链路，再做交互层优化。

### 4.4 DAP 协议 — 产品核心护城河

**DTS Agent Protocol (DAP)** 是 DTS 的核心护城河，定义了 AI Agent 如何理解物理世界并执行决策。

```
Layer 5: Audit Protocol        — 决策可追溯性保障
Layer 4: Security Protocol     — 身份与权限在 Agent 间传递
Layer 3: Intent Protocol       — 自然语言 → 结构化意图 → 执行计划
Layer 2: Skill Protocol        — AI 技能的定义、注册、调用、编排
Layer 1: Ontology Protocol     — 物理世界 → 数字映射的标准格式
Layer 0: Transport             — gRPC (control) + Kafka (data) + SSE (streaming)
```

**为什么 DAP 是护城河**：

- 代码可以被重写，**协议定义的生态不能**
- 每个行业 Pack 遵循 DAP 接入，形成**网络效应**
- 客户的 Ontology 和 Skill 资产建立在 DAP 之上，**迁移成本 = 丢失核心竞争力**
- 类比：Android 的护城河不是 AOSP 代码，而是 Google Play 生态协议。Palantir 的护城河不是 Foundry 代码，而是 Ontology 数据模型标准

**协议版本规划**：

| 版本 | 能力 | 时间 |
|------|------|------|
| DAP v1 | 结构化 Ontology + Skill + Intent + Security + Audit | 当前 |
| DAP v2 | 多模态 Intent（语音/视觉输入），多模态 Ontology 表征 | 12-18 月后 |
| DAP v3 | 跨组织 Agent 协作（Agent-to-Agent），世界模型 | 18-24 月后 |

## 5. 知识战略（Polanyi + LeCun）

### 5.1 哲学基础：为什么 DTS 对"知识"的理解不同

传统数据平台的隐含假设是实证主义知识观：知识 = 可量化、可存储、可检索的数据。这个假设在"数据分析"场景下成立，但在"企业决策"场景下根本性失败——因为真正驱动决策的知识，大部分**不是数据**。

DTS 的认识论基础来自两个源头：

- **Michael Polanyi（波兰尼）**：缄默知识理论——"We know more than we can tell"（我们知道的比我们能说出来的多）
- **Yann LeCun（2026 论文）**：Beyond Language Modeling——语言只是物理世界的投影，视觉和行为数据承载了远比语言更丰富的知识

**DTS 的核心洞察**：企业最有价值的知识——老师傅听机器声音判断故障、调度员凭直觉调整排产、质量工程师一眼看出不良品——这些缄默知识无法被传统系统"存储"，但可以被 DTS 通过 Ontology 建模 + AI 对话逐步"解蔽"（unconceal）。

### 5.2 五层知识金字塔

```
Layer 4: 世界模型 (World Model)
  Agent 能预测操作后果: State + Action → Next State
  LeCun 发现: 通用预训练 + 1% 领域数据即可涌现
  ──────────────────────────────────────────────
Layer 3: 隐性模式 (Tacit Patterns)
  专家没说但一直在用的规则
  从多模态行为数据中统计推断
  Polanyi: subsidiary awareness
  ──────────────────────────────────────────────
Layer 2: 潜在知识 (Latent Knowledge)
  专家能说但没人问过的判断逻辑
  Gen-Ba 碎片化提取 (决策瞬间自然追问)
  ──────────────────────────────────────────────
Layer 1: 显性结构 (Explicit Structure)
  数据字典、行业标准、系统表结构
  传统 Ontology 建模 — 必要但不充分
  ──────────────────────────────────────────────
Layer 0: 多模态感知 (Multimodal Perception)
  视觉 + 声音 + 时序 + 文本
  身体化知识 (embodied knowledge)
```

传统 BI/数据中台只覆盖 Layer 1（显性结构）。DTS 的目标是逐步覆盖 Layer 0-4，最终构建企业对物理世界的完整认知。

### 5.3 KnowHow 提取流水线 — 超越 Nonaka SECI

传统的知识管理（Nonaka 的 SECI 模型）失败了——让专家填表格、写文档，结果全是废话，没人看。DTS 采用 Gen-Ba 碎片策略，在决策瞬间自然追问，而非系统性访谈。

| 阶段 | 方法 | 关键原则 |
|------|------|---------|
| Phase 1: 影子观察 | AI Agent 静默观察专家日常决策 | 不干扰工作流 |
| Phase 2: 碎片外化 | 决策瞬间自然追问："您刚才为什么选择提前维保？" | 容许不完整、容许矛盾、标记置信度 |
| Phase 3: 松散组合 | LLM 将碎片聚类、去矛盾、补全，生成 KnowHow 草稿 | 不追求完美，追求"够用的近似" |
| Phase 4: 协作内化 | KnowHow 转为 Skill，与专家并行运行，持续校正 | 通过 evaluation suite 量化准确率 |

**与 SECI 的关键区别**：SECI 假设隐性知识能被"完全转化"为显性知识，这是认识论幻觉。DTS 承认隐性知识有不可消除的缄默维度，只追求"够用的近似"，剩余不确定性由 HITL（人工介入）覆盖——这正是铁律 #1 的认识论基础。

### 5.4 世界模型战略（1% 领域数据涌现）

LeCun 2026 年论文的关键发现：通用预训练模型 + 仅 1% 领域数据的微调，即可在特定领域涌现出显著的理解能力。这为 DTS 提供了重要的技术路线支撑：

- **短期**（DAP v1）：纯结构化 Ontology，专注 Layer 1-2 的显性和潜在知识
- **中期**（DAP v2）：多模态 Ontology 扩展，覆盖 Layer 0 的感知数据和 Layer 3 的隐性模式
- **远期**（DAP v3）：世界模型，Agent 能在模型中"模拟运行"操作后果，实现 Layer 4 的预测能力

## 6. 产品护城河

### 6.1 DAP 协议的网络效应

DAP 不是一个技术规范，而是一个**生态标准**。类比：

- HTTP 定义了浏览器与服务器的交互 → 催生了整个 Web 生态
- Android 的 Activity/Service 协议 → 催生了 Google Play 300 万应用
- **DAP 定义了 AI Agent 与企业物理世界的交互 → 催生 DTS AppPack 生态**

每一个新的行业 Pack 遵循 DAP 接入，都在强化这个生态：更多 Pack → 更多客户 → 更多开发者 → 更多 Pack。网络效应一旦形成，后来者即使代码写得更好，也无法复制生态。

### 6.2 KnowHow 资产锁定

DTS 的商业模式本质是**知识资产平台**，而非工具租赁。客户使用 DTS 3 年后，系统中沉淀了：

- 数百条行业决策规则（Ontology 中的 Action + Risk Level）
- 数十个 AI 技能（Skill 中编码的专家经验）
- 数千条 KnowHow 碎片（从决策瞬间提取的隐性知识）
- 完整的决策审计轨迹（组织级决策模式的数据资产）

**切换成本 = 丢失核心竞争力**。这不是技术锁定（开放的 DAP 规范支持导出），而是**知识资产的不可替代性**——在别的平台上重建这些知识资产需要再花 3 年。

### 6.3 行业深度壁垒 — Level 0-3 深度光谱

```
Level 0    通用数据平台             ← 传统大数据平台停在这里
Level 1    行业知识插件             ← 含 Ontology + Tools + Prompts 的 AppPack
Level 2    垂直应用模块             ← "在 AppPack 里做 MES 决策层"
Level 3    独立垂直产品             ← 某个 AppPack 足够厚，可独立部署
```

DTS 的策略是"做项目炼认知，不做项目卖产品"：

- AppPack 只装**决策层精华**（行业知识的 20%），不做完整应用
- 你已有的 MES 管流程，DTS 管决策
- 每个项目强制做"沉淀物/交付物分离"，AppPack 是项目的副产品
- **复用率 > 60% = 行业理解到位**；< 40% = 还没抓住行业本质

### 6.4 与 Palantir 的六维对比

| 维度 | Palantir Foundry/AIP | DTS v3.0 |
|------|---------------------|----------|
| **知识模型** | Ontology（封闭，客户无法迁移） | DAP Ontology（开放协议，可导出迁移） |
| **AI 架构** | AIP 内建 LLM 编排 | Ontology + Intent + Agent 三层解耦 |
| **行业扩展** | 代码级定制，需 Palantir 工程师 | AppPack 全栈插件，行业专家可通过 AI 对话创建 |
| **部署模式** | SaaS 为主 + 政府私有化 | 私有化为主，支持离线/信创/一体机 |
| **价格** | $1M+/年起步 | 50-500 万 RMB/年，渐进导入 |
| **生态模式** | 封闭平台 | 开放 AppPack 市场 + KnowHow 交易 |

**核心差异**：Palantir 的模式是"重咨询 + 代码定制"，客单价高但扩展慢。DTS 的模式是"开放协议 + 知识资产市场"，通过 AppPack 生态实现规模化扩展。

```

**第一层：平台基础费** — 按部署规模定价，刻意鼓励全员使用

- 标准版 30-80 万/年（中型企业，5 个以内数据源）
- 企业版 100-300 万/年（大型企业，无限数据源）
- 不按人头收费——用 10 人和用 1000 人一个价，使用越多 → 知识沉淀越多 → 越离不开

**第二层：AppPack 市场** — 行业能力包按需购买

- 每个 Pack 独立定价，按场景价值而非功能数量
- 行业包 10-50 万/个/年，高价值场景 80-100 万
- 客户/ISV/行业专家开发的 Pack 上架销售，平台抽成 30%

**第三层：KnowHow 交易** — 知识资产的 App Store

- 行业一线专家通过 AI 提取经验 → 结构化为 KnowHow → 发布到市场
- 其他企业购买使用，专家持续获得分润
- 这是"越用越值钱"飞轮的核心驱动力——**不是卖工具，而是让知识流动产生价值**

### 7.2 "做项目炼认知，不做项目卖产品"飞轮

```
接行业项目 → 真实场景锻炼
    ↓
获得缄默知识 → 理解行业决策的真正逻辑
    ↓
提炼为 AppPack → 只留决策精华(20%)，交付物留给客户
    ↓
用 AppPack 接下一个同行业客户 → 交付更快、成本更低
    ↓
AppPack 持续迭代 → 复用率越来越高
    ↓
达到复用率 > 60% → 行业 AppPack 定型 → 开放市场销售
    ↓
（循环 — 进入下一个行业）
```

**关键纪律**：每个项目强制做"沉淀物/交付物分离"——行业 Ontology、决策 Skill、对话 Prompt 进 AppPack（沉淀物），CRUD、审批流、定制报表留在客户项目（交付物）。不让自己滑成应用厂商。

### 7.3 AppPack 只含决策层精华（20%）

用波兰尼的缄默知识框架分析行业知识的三层结构：

| 层级 | 内容 | 谁做 | 在 DTS 中 |
|------|------|------|----------|
| **决策层（最值钱）** | "这批料该不该让步接收""产线怎么排" | DTS AppPack | Ontology + Skill + Prompt |
| 流程层（工作量大） | 工单状态机、审批流、BOM 展开 | 传统 MES/ERP | 不做，通过 Connector 对接 |
| 数据层（无差异化） | 工序记录、SPC 数据、报工数据 | CRUD，谁都能做 | 不做，读客户已有数据 |

**定位**：你已有的 MES/ERP 管流程和数据，DTS 管决策。AppPack 只装最上面那 20% 最值钱、最难复制的决策知识。

### 7.4 客户分层

| 层级 | 客户画像 | 客单价 | 交付模式 |
|------|---------|--------|---------|
| **旗舰客户** | 央企/大型国企（电网、石油、航天） | 200-500 万/年 | 离线私有化 + 深度定制 + 驻场 |

| **中小客户** | 数据驱动型中小企业 | 10-30 万/年 | 轻量私有化/边缘一体机 |

### 8.2 5 天 POV 方法论

DTS 不卖 PPT，卖体验。用客户自己的数据，5 天跑出真实结果：

```
Day 1: 环境部署（1小时内跑起来）+ 接入客户 1-2 个核心数据源

**核心承诺**：不搬数据（JDBC 直连），不改系统（与现有 IT 共存），不绑架客户（任何阶段可无损退出）。

### 8.3 标杆客户选择

**Phase 1（0-12 月）— 深耕 2 个行业**：

- **能源/电网**：已有行业认知积累，政策驱动（新型电力系统数字化），客单价高（200-500 万），决策场景丰富（调度优化/设备维保/负荷预测）
- **科研院所**：需求明确（实验数据管理 → 智能分析 → 论文辅助），决策链短（课题组即可决策），场景标准化程度高

**Phase 2（12-24 月）— 扩展 3 个行业**：

- 先进制造（质量决策/排产优化/供应链）
- 医药（研发数据管理/临床决策/合规）
- 新能源（风电/光伏/储能运维决策）

### 8.4 行业切入的"投资保护"策略

DTS 对客户的核心承诺：**"你之前的投资解决了'有没有数据'的问题。DTS 解决'数据能不能做决策'的问题。不是替换，是让已有投资增值。"**

| 客户已有 | DTS 做法 |
|---------|---------|
| 数据湖/数仓 | 不搬数据，JDBC 直连，SQL 下推执行 |
| BI 工具（帆软/Tableau） | 可调用其报表，也可独立生成 |
| ETL 工具（Airflow/DataX） | 可编排触发，不替换 |
| MES/ERP | 通过 Connector 读取数据，DTS 管决策 |
| 数据字典/标准 | 自动导入，构建 Ontology 基础 |

### 9.2 Manus 的启示与警示

Manus 8 个月达到 $100M ARR，被 Meta $2B 收购。这个案例说明：

**正面信号**：AI Agent 市场需求真实且增长迅猛。

**反面教训**：
- 通用 Agent 没有行业壁垒 → 最终被大模型厂商吞掉（被 Meta 收购正是此逻辑）

- **做"万能助手"是死路**——模型厂商自己就能做

**DTS 的定位**：不做通用 Agent（那是模型厂商的菜），做**行业深度 + 知识沉淀 + 安全合规**——这是模型厂商做不了、不愿做的脏活累活。

### 9.3 与 Palantir 的深度对比

Palantir 是 DTS 最重要的参照系。两者在产品理念上高度一致（Ontology 驱动的 AI 决策平台），但在商业模式和市场策略上有本质区别：

**Palantir 的局限**：
- **重咨询模式**：每个客户需要 Palantir 工程师驻场数月定制，毛利率受限
- **封闭生态**：没有 AppPack 市场，行业扩展依赖自身团队
- **价格壁垒**：$1M+/年起步，只服务大型企业和政府
- **不服务中国**：地缘政治限制，无信创兼容

**DTS 的机会**：
- **开放生态**：AppPack 市场让 ISV 和行业专家都能创造价值
- **自动化知识提取**：AI 对话提取隐性知识 vs Palantir 的代码级定制
- **中国市场**：离线部署、信创兼容、本土化服务
- **知识交易**：KnowHow 市场是 Palantir 没有的全新维度

### D. DAP 协议摘要

**Layer 1 — Ontology Protocol**：定义 AI Agent 如何理解客户的物理世界。ObjectType（业务对象）、Relationship（关系）、Metric（指标）三种核心资源，声明式定义，数据安全内置，行业命名空间隔离。

**Layer 2 — Skill Protocol**：定义 AI 能力的标准接口。Skill Descriptor 声明技能元数据（类型/权限/输入输出/评测基线），gRPC 标准接口（Execute/Describe/Validate），流式响应 + 审计内嵌。

**Layer 3 — Intent Protocol**：定义自然语言如何转化为结构化决策意图。Intent 结构包含原始输入、解析目标、执行计划（DAG），支持条件执行和风险分级。

**Layer 4 — Security Protocol**：定义 Agent 间的身份传递。JWT 从 Keycloak → dts-gateway → gRPC metadata → 每个服务，零信任校验。数据安全在 Agent 链路内实时拦截，AppPack 无特权。

**Layer 5 — Audit Protocol**：定义 AI 决策的完整可追溯性。CloudEvents 标准格式，10 种事件类型覆盖全链路，append-only 不可篡改，token 成本追踪支持用量计费。

### E. 五条铁律

1. **Human Override Guarantee** — 删掉 AI，系统仍 100% 可用。任何步骤可人工接管。
2. **Security Cannot Be Bypassed** — 所有请求必经 dts-gateway，无旁路直连，AppPack 无特权。
3. **Data Security Is DNA** — 所有数据出口必经 dts-data-security，AI 只看授权数据。
4. **Full Audit Trail** — 人和 AI 的操作 → Kafka → dts-audit-log，append-only 不可篡改。
5. **Capability Before Interface** — API-first，先有完整 API + 测试，再做前端。
